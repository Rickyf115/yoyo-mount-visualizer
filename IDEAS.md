# Ideas parking lot

Out-of-scope ideas logged here instead of expanding a session (see CLAUDE.md rules).

- **More play styles.** The engine currently assumes 1A (single unresponsive yo-yo, string attached to the throwhand). Future: 5A (counterweight — the loop end becomes a free mass, so `loop` stops being a fixed anchor), 3A (two yo-yos/two loops), 2A/looping (responsive, mounts barely apply), 4A/offstring (no axle winding — the terminal `axle` contact assumption breaks). Each style likely means relaxing a traversal invariant, so gate them behind a `style` field when the time comes.
- **Single-element route for trapeze → double or nothing.** The element set reaches double or nothing from trapeze via two passes (through the mid-swing state), but the Cascade fixture encodes it as *one* unpinned step, and step inference (Session 6 integrity check) matches one element per step. Either add a composite "swing around both hands" element, or split such fixture steps — decide when the integrity test lands.
- **Elements over mounts with crossings.** Every element currently preconditions on `crossings: []` (no fixture has any). Inserting/removing contacts shifts segment indices, so crossing-aware transforms need index remapping — build it when the first crossing fixture appears.
- **Constant string length.** The rope's rest length currently retargets to each layout's polyline length (lerped through transitions). A real string is constant-length — modeling that means layouts must budget slack (the free tail below the axle winding) explicitly.
- **Swing direction should follow spin.** Arc direction is currently chosen by "reach the apex the short way"; physically the yo-yo swings in the direction of its momentum, which follows spin sense. Needs the finer spin model below.
- **Rope-driven yo-yo.** The yo-yo currently rides a scripted arc and drags the rope. Letting the yo-yo be a mass at the rope's end (with the arc as a guide force) would give natural overshoot and settle on landing.
- **Finer spin model.** `spin: front | side` is player-relative and binary. If it proves too coarse (e.g. distinguishing same-plane opposite-sign spin for lefty/righty interactions, or spin relative to the mounted string segment — which determines whether a given roll winds or unwinds), refine the vocabulary then; it's another deliberate hash change.
- **Mirror equivalence.** Canonicalization preserves sidedness, so a left-handed player's trapeze hashes differently. A `mirrorHash` (canonicalize after swapping L/R and flipping direction/throw) would let search dedupe mirror twins without collapsing them.
- **Loop position.** The throwhand slipknot is assumed on the middle finger and carries no digit. If tricks that re-seat the loop matter, give `loop` an optional digit.
- **Where the string end sits.** Some mounts are distinguished by how much tail hangs between the last wrap and the axle (e.g. wrist mounts with big loops). If topology alone under-distinguishes, consider annotating segment slack classes.
